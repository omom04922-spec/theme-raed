# Scope of this security policy
# This security.txt file applies to:
# - Salla Theme Raed (https://github.com/omom04922-spec/theme-raed)
# - All related theme files and components
# - Theme configuration and settings
# - Theme assets and resources
# - Theme documentation and examples

# What to report:
# - Cross-Site Scripting (XSS) vulnerabilities
# - Cross-Site Request Forgery (CSRF) vulnerabilities
# - SQL Injection vulnerabilities
# - Authentication and authorization bypasses
# - Sensitive data exposure
# - Server-side request forgery (SSRF)
# - Remote code execution (RCE)
# - Local file inclusion (LFI)
# - Directory traversal vulnerabilities
# - Insecure direct object references
# - Security misconfigurations
# - Insecure cryptographic implementations
# - Business logic vulnerabilities
# - Privacy violations
# - GDPR compliance issues
# - Accessibility security issues
# - Mobile security vulnerabilities
# - API security vulnerabilities
# - Third-party integration security issues

# What NOT to report:
# - Issues in third-party dependencies (report to the respective maintainers)
# - Social engineering attacks
# - Physical security issues
# - Denial of Service (DoS) attacks
# - Spam or phishing attempts
# - Issues requiring physical access to devices
# - Issues in outdated versions (please test on the latest version)
# - Issues that require user interaction with malicious content
# - Self-XSS that cannot be used to attack other users
# - Missing security headers without demonstrable impact
# - Clickjacking without demonstrable impact
# - CSRF on forms that are not state-changing
# - Missing rate limiting without demonstrable impact
# - Theoretical vulnerabilities without proof of concept

# Reporting guidelines:
# 1. Provide a clear description of the vulnerability
# 2. Include steps to reproduce the issue
# 3. Provide proof of concept code or screenshots
# 4. Specify the affected versions
# 5. Include the potential impact of the vulnerability
# 6. Suggest possible fixes if you have any
# 7. Do not access or modify data that doesn't belong to you
# 8. Do not perform actions that could harm the service or users
# 9. Do not publicly disclose the vulnerability until we've had a chance to fix it
# 10. Be patient - we'll respond as quickly as possible

# Response timeline:
# - Initial response: Within 24-48 hours
# - Vulnerability assessment: Within 1 week
# - Fix development: Depends on severity and complexity
# - Fix deployment: As soon as possible after development
# - Public disclosure: After fix is deployed and users have had time to update

# Severity classification:
# Critical: Immediate threat to user data or system integrity
# High: Significant security risk with potential for data breach
# Medium: Moderate security risk with limited impact
# Low: Minor security issue with minimal impact
# Informational: Security-related observation without immediate risk

# Reward program:
# We appreciate security researchers who help us improve our security.
# While we don't currently offer monetary rewards, we do provide:
# - Public acknowledgment (if desired)
# - Direct communication with our security team
# - Updates on fix progress
# - Invitation to test fixes before public release

# Legal safe harbor:
# We will not pursue legal action against security researchers who:
# - Follow responsible disclosure practices
# - Do not access or modify user data
# - Do not perform actions that could harm our services
# - Report vulnerabilities in good faith
# - Comply with applicable laws and regulations
